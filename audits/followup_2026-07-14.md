## F-1. Coupled models are EXEMPT from reference-integrity checking in every binding

**Status:** RESOLVED 2026-07-16 (branch `fix/conformance-2026-07-15`). The blanket exemption was
narrowed in all five bindings and the arbitrary-placeholder fixtures were migrated to `_var`. A
coupled model — operator-composed or a coupling target — is now reference-checked against the
DOCUMENT-WIDE declared names ∪ `_var` (composition merges the participating systems' scopes, so a
cross-system bare reference such as the end-to-end atmosphere/land fixtures' `air_temperature`
resolves), while a name declared NOWHERE is now caught as `undefined_variable`. This is Python's
long-standing behaviour (`global_symbols` + unconditional `_var`); Julia/Go/Rust/TS were brought to
match it. Verified: all five bindings accept the migrated corpus, reject an injected typo inside a
coupled model, and `./scripts/test-conformance.sh` passes end-to-end. Renamed the bare `u`
placeholder to `_var` in `tests/valid/minimal_chemistry.esm` (`Advection`),
`tests/valid/metadata_inheritance_coupled.esm` (`AdvectionTransport`),
`tests/coupling/coupling_resolution_edge_cases.esm` (`PlaceholderTestSystem`),
`tests/version_compatibility/comprehensive_compatibility_test.esm` (`Transport`) and
`tests/scoping/bare_reference_resolution.esm` (`Transport`). A pre-existing Rust bug the exemption
had masked — `makearray` grid indices not bound across sibling `values` — was fixed by collecting
bound symbols across the whole expression (matching Go/TS). The original analysis is retained below.

**Original status (open):** Behavioural change across five bindings; deferred rather than opened as another wave.

### The gap

`esm-spec.md` §6.4 mandates **`_var`** as *the* operator-model placeholder — a **reserved** name,
substituted with each matching state variable of the target system at `operator_compose` time. Being
reserved is the whole point: a checker can recognise it locally, without resolving the composition.

The corpus does not honour that mandate. The flagship fixture **`tests/valid/minimal_chemistry.esm`**
declares an operator model `Advection` whose `variables` map contains only `u_wind` and `v_wind`, and
whose equations reference a bare **`u`** — an operator placeholder the author named arbitrarily, and
which is declared nowhere.

### Why that one fixture forces a blanket exemption

`u` is an ordinary identifier. Nothing about it is distinguishable, by any local analysis, from a
typo — which is exactly the false-negative class that §4.9.5 reference integrity exists to kill. A
checker looking at `Advection` in isolation sees an undefined variable and cannot know it is a
placeholder.

So every binding is forced into the blunt instrument: **exempt operator-composed models, and coupling
targets, from reference checking entirely.** The consequence is the part worth writing down —

> **Coupled models are currently NOT reference-checked in any of the five bindings.**
> An undefined variable inside an operator-composed model is silently accepted today. The (h) work of
> 2026-07-14 closed reference integrity across all thirteen expression-bearing fields, but coupled
> models sit behind this exemption and see none of it.

(The TypeScript agent attempted to remove the exemption in this wave, broke `minimal_chemistry.esm`,
and correctly backed the change out. The fixture — not the checker — is what blocks it.)

### A second fixture with the same defect (found 2026-07-14 by the Python binding)

`tests/coupling/coupling_resolution_edge_cases.esm` → model **`PlaceholderTestSystem`**, which is the
operator model of `coupling[7]`, an `operator_compose`. Its equations read a bare **`u`** that is
declared nowhere (it declares only `normal_var`) — the same arbitrary-placeholder defect as
`minimal_chemistry.esm`.

Worse, that entry's own description reads *"EDGE CASE 8: System with variable literally named `_var` —
disambiguation required"*. The fixture was **written to test `_var`** and then spelled the placeholder
`u`, so it does not exercise the case it names, and it declares no `_var` variable for the
disambiguation it claims to check. Renaming `u` → `_var` fixes the reference defect *and* makes the
fixture test what it says. (Deferred with F-1 rather than fixed piecemeal, because the rename only pays
off once the blanket exemption is narrowed.)

### Proposed remedy

Migrate the fixtures to the spec-mandated placeholder:

1. In `tests/valid/minimal_chemistry.esm` and `tests/coupling/coupling_resolution_edge_cases.esm`
   (`PlaceholderTestSystem`) — and any other sibling that names an operator placeholder arbitrarily;
   `full_coupled.esm` already uses `_var` and is the model to copy — rename the bare `u` to **`_var`**.
2. Narrow the exemption in each binding from *"skip reference checking for coupled models"* to
   *"`_var` is in scope wherever a state variable is legal, in a model that is operator-composed or is
   a coupling target"* — which is what §6.4 already says, and what the §4.9.5 in-scope table
   (`esm-spec.md`, the `_var` row) already pins.
3. Coupled models then get **actual** reference-integrity checking, with a one-name exemption instead
   of a whole-model one.

### Why it was deferred

The rename is a behavioural change that lands simultaneously in five bindings, and the 2026-07-14 wave
was already past the point where another cross-language change could be absorbed safely. It is a clean,
self-contained follow-up: one fixture edit, then a one-line narrowing of the exemption in each binding.

**Do not "fix" this by relaxing §6.4 to permit arbitrary placeholder names.** An arbitrary placeholder
is unrecognisable to a local checker by construction, so permitting it would make the blanket exemption
permanent and leave coupled models unchecked forever. The reserved name is what buys the checking back.

## F-2. `validate()` takes no `basePath`, so relative-ref pins cannot be satisfied

**Status:** open, TypeScript-side (`pkg/earthsci-ast-ts`). Blocks one pin today; blocks many once F4 lands.

TypeScript's `validate(data: string | object)` accepts no `basePath`, so it cannot open a file named by
a relative `subsystems[].ref` or `expression_template_imports` entry. It therefore reports
`unresolved_subsystem_ref` ("file does not exist") where the correct finding needs the file to be read.

`tests/invalid/subsystem_ref_ambiguous.esm` is the live case: its ref `../scoping/nested_subsystems.esm`
**does** exist and **does** declare two top-level systems, so the file is genuinely
`ambiguous_subsystem_ref` — the pin is correct. TypeScript cannot currently emit that code for it.

This is invisible today only because the conformance harness asserts `is_valid === false` and not the
code. **The moment F4 compares `(code, path)`, every subsystem-ref and template-import pin becomes
unsatisfiable in TypeScript.** Remedy: give `validate()` a `basePath` option (as `load()` already has),
or mark the affected pins `resolver_only`. The former is correct; the latter merely hides it.

## The conformance harness is RED on merge to main — this is intentional and tracked

As of the 2026-07-14 integration merge, `./scripts/test-conformance.sh` **exits non-zero.** This is
**by design**: the harness was rebuilt in this wave so that it *can* fail (previously it printed
"completed successfully!" and exited 0 with three bindings failing — audit C1/C2/F4/F5). It now
reports **1376 true cross-language failures**, none of them regressions — they are pre-existing bugs
the old harness was structurally incapable of seeing. **All five per-binding suites are GREEN**
(Julia 9372 · TS 1500 +346 editor · Python 1335 · Rust 850 · Go clean); it is only the
cross-language conformance comparison that is red, and correctly so.

The 1376 decompose by root cause below. `CLAUDE.md` says implementations "must pass
`./scripts/test-conformance.sh`" — that becomes true again only as these are worked off; until then
a red harness is the honest state and must not be silenced by weakening an assertion.

### F-3. Schema errors have five incompatible wire formats — 317 pin failures
Each binding reports JSON Schema violations differently: **TS** = JSON Pointer + the JSON Schema
keyword (correct); **Go** = dotted `(root).coupling.0` + a non-standard vocabulary; **Julia** = JSON
Pointer but only the enclosing `oneOf`; **Python** = one `$`-rooted blob; **Rust** = one `""` blob.
Remedy (needs a spec decision, then a pass across four bindings): standardise on **JSON Pointer + the
JSON Schema keyword vocabulary** — what the JSON Schema spec's own output format prescribes, and what
TS already emits. Update `CONFORMANCE_SPEC.md` §7.1 accordingly.

### F-4. Display/rendering divergence — 1015 failures
- **Go has no operator layer** (233, the single largest root cause): every operator prints as a
  function call — `and(x > 0, x < 10)` where the contract requires `x > 0 ∧ x < 10`; `=(x,0)` for `==`.
  Go only ever consulted `structural_ops.json`. Needs a real operator-rendering layer.
- **Julia** LaTeX subscripts `O_{3}` vs `O_3` (64); **Rust** chemical subscripts/charge + LaTeX `_`
  escaping (85); **Python** `&&`/`||`/`!` in ascii + untransliterated Greek (`sin(θ)` vs `sin(theta)`);
  **TS** charge superscript, `\text` vs `\mathrm`, scientific notation.

### F-5. Integer-vs-float canonicalization — 44/50 property-corpus expressions + 3 substitution
**Go floats everything** (`0` → `0.0`, `[9,6]` → `[9.0,6.0]`) — ~42 of 44; Julia/Python/Rust float in
scattered positions. One canonical-number rule, applied everywhere, closes both gates.

### F-6. Five `tests/invalid` fixtures need a static check no binding implements — 25 outcome failures
`float_join_key`, `null_in_key_column`, `domain_unit_mismatch_identity`, `continuous_relational_node`,
`undeclared_from_name`. Each is decidable **statically from its single document** (compare two declared
literals / two declared `units`) — `validate()` already does exactly this for
`units_coupling_mismatch.esm`. They are five real binding gaps, not resolver-only. Promote to
structural pins once implemented.

### F-7. SIX display goldens where all five bindings are wrong TOGETHER — and the harness mislabels them
`a^b^c` (must render `(a^b)^c` — `^` is right-associative, so the bindings' output reads back as
`a^(b^c)`, a semantic error); `∂(x + y)/∂t` (bindings drop the parens); `∞`/`−∞` (bindings emit literal
`Infinity`/`-Infinity`); `Ca²⁺` (bindings emit `Ca₂+` — the 2 is the **charge**, not an atom count);
`\text{NaN}`. The golden is RIGHT; every binding is WRONG. **The harness currently reports these as 12
"CORPUS failures — the pin has drifted"** via an unsound *unanimity ⇒ pin-is-wrong* heuristic. Fix the
harness's routing (unanimity is not evidence the golden is wrong), and fix the six renderers.

### F-8. Rust's `validate_complete(&str)` is base-path-unaware — 2 valid fixtures rejected (Rust 80/82)
Same defect as F-2 but Rust-side and load-affecting: `validate_complete` re-`load`s from the raw string,
so relative §4.7 / §9.7 refs resolve against the process CWD. The producer calls `load_path` correctly
then discards it. Rust's own suite is green because it uses `load_path`; only the recursive conformance
sweep exposes it. Remedy: a `base_path` parameter (Python already did this).

### F-9. TS emits no structural finding for `subsystem_ref_ambiguous`/`subsystem_ref_not_found`
A genuine error-identity false negative (distinct from F-2's path/code granularity): TS rejects the
files but emits no structural finding object the pin can match. 8 TS pin mismatches total, of which the
root-path `$` vs `""` convention is the other 4 (settle in F-3).

### F-10. The Julia PDE-simulation producer cannot bootstrap — 2 harness stages
`scripts/pde_sim_adapter/Project.toml` declares the unregistered, local-only `EarthSciIO`, but the
bootstrap only `Pkg.develop`s EarthSciAST → `expected package EarthSciIO to be registered`. Pre-existing
(reproduces on `fix/harness` and with the default depot), **not** a merge regression. Remedy: `develop`
EarthSciIO from its local path in the adapter bootstrap, or gate the two PDE stages behind its presence.

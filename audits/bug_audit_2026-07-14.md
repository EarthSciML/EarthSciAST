# EarthSciAST — Repo-wide Bug Audit

**Date:** 2026-07-14
**Scope:** all five bindings (Julia, Python, Rust, Go, TypeScript + editor), the shared
fixture corpus, and the cross-language conformance harness.
**Status:** review-only. Every finding below was substantiated against the current tree
(most reproduced empirically); no code was changed.

---

## Executive summary

The individual bindings are in better shape than the *system* is. Each binding has a
respectable test suite that passes (Rust 806, TS 1301, editor 346, Python 1220, Go green),
and the recent Julia CSE/cadence fix pass holds up under attack — I tried to break all seven
fixes from `julia_iip_cse_cadence_audit_2026-07-14.md` and could not break any.

What is broken is the layer that is supposed to hold the five bindings to *one* specification.
Three findings define the shape of this audit:

1. **Dimensional-unit validation is broken in four of the five bindings, in four different
   ways.** Julia never calls it; Python's core predicate returns `true` for every input; TS's
   `^` ignores its exponent and its unit parser cannot handle parentheses; Rust rejects the
   integer exponent that is the natural spelling of `L^2`. Only Go is correct. The bindings do
   not merely have bugs here — they *disagree with each other* about which files are valid.

2. **The bindings have quietly diverged on what "valid" means, and it is not a small
   divergence.** TypeScript reports `is_valid: false` for **22 of the 82** `tests/valid/`
   fixtures; Go reports invalid for **41 of 82**. Julia *accepts* 15 of the `tests/invalid/`
   fixtures. No test in any binding asserts that `tests/valid/**` is valid.

3. **The conformance harness is structurally incapable of catching any of that.**
   `./scripts/test-conformance.sh` — the command `CLAUDE.md` tells developers to run — prints
   "completed successfully!" and **exits 0 with Julia, Rust, and Go all failing.** The
   comparator passes at a 70% consistency score, two of its three compared categories are
   vacuous, it never checks a fixture against its *expected* outcome (only languages against
   each other), and it sweeps `tests/valid`/`tests/invalid` non-recursively, missing 69
   fixtures.

Underneath these sit two recurring root causes worth naming, because each one explains a
cluster of findings rather than a single bug:

- **Hand-rolled walkers that bypass the one-walker keystone.** Go and TS both have a correct,
  field-preserving `mapExprChildren` / `mapChildren` — and then hand-roll `freeVariables`,
  `contains`, `substitute`, `flatten`, and the graph builder to walk `args` only. Every
  expression-bearing sidecar field (`expr`, `axes`, `lower`, `upper`, `filter`, `key`,
  `values`, `bindings`, `regions`) is invisible to them. This is the direct cause of G1–G4,
  G11, T7–T10 — including a Go bug that silently **corrupts the model** (G2).
- **Skip mechanisms that are permanently green.** Python's imperative `pytest.xfail()` makes an
  XPASS structurally impossible (13 of its 17 skipped entries now pass and nobody noticed);
  Julia's `@test_skip false` is a test that can never pass; the `ts_skip` and closed-function
  skip hatches are dead code; and every tracking bead ID cited in these skips (`gt-qgui`,
  `gt-i7e1`, `esm-mvc`, `esm-y3n`, `gt-pcj5`, ~20 in total) **does not exist** in the beads
  database, so no skip can ever expire.

**The good news, stated plainly:** the recent Julia CSE work is sound, Rust's vectorized
bail-out discipline and buffer-ownership rules are airtight, Go's `mapExprChildren` and closed
functions are correct, TS's lossless numeric round-trip is byte-idempotent across all 82 valid
fixtures, and TS's fail-closed canonicalization allow-list is the right design. The bugs are
concentrated, not diffuse.

## 1. Units and dimensional analysis — broken in 4 of 5 bindings

This is the single most damaged subsystem in the repo, and the damage is *independent* in each
binding, which is why no cross-binding test caught it.

### C5 — Python: the dimensional check is dead code
`pkg/earthsci-ast-py/src/earthsci_ast/units.py:387-402`

`_dimensions_compatible` builds `self.ureg.Quantity(1.0, dim1)` from a **dimensionality**
container (`[length]`) rather than a unit. `q1.to(q2.units)` then trips pint's
`assert len(names) == 1` in `_is_multiplicative` — because `[length]` is not a defined unit
name — raising a bare `AssertionError`, which the handler
`except (pint.PintError, AssertionError): return True` swallows.

The function therefore **returns `True` for every input pair**:

```
_dimensions_compatible([length], [time])          -> True
_dimensions_compatible([pressure], dimensionless) -> True
```

A model with `x: "m"`, `tt: "s"`, equation `x = x + tt` yields
`is_valid=True, errors=[], warnings=[]`. Every mismatch path in the module — including
`_get_expr_node_dimension`'s `raise ValueError("Incompatible dimensions in +")` — is
unreachable. `validate_units` / `UnitValidator` are public exports.

The in-code comment claims the `AssertionError` only leaks for offset units like
`[temperature]`. That is wrong: it leaks for *every* bracketed dimension. The correct pattern
already exists at `structural_checks.py:382` (`q1.dimensionality == q2.dimensionality`).

**Why no test caught it:** `tests/test_unit_validation.py:542-550` asserts only that validation
"completes without raising"; `:627-663` assert the *absence of false positives*. Nothing in the
suite asserts a true positive.

Two more Python unit bugs sit behind this one and will surface the moment it is fixed:

- **`units.py:341-372`** — `_get_expr_node_dimension` filters `None` dimensions out of
  `valid_dims`, then indexes `/` and `^` operands **positionally** as though nothing was
  removed. `{"op":"/","args":["unknown_x","t"]}` with `t: "s"` reports dimension `[time]` —
  the exact *inverse* of the only defensible answer. Correct behavior is `None`.
- **`units.py:384-385`** — the catch-all `return dimensionless` also swallows `min`, `max`,
  `abs`, `sign`, `ifelse`, and `D`, all of which **preserve** their operands' dimension.
  `max(P1, P2)` with both in Pa reports dimensionless. Conversely nothing checks that
  `sin`/`exp`/`log` *arguments* are dimensionless, so `sin(L)` with `L` in metres passes.

### J1 — Julia: validation never calls the units engine, and the engine can't fail
`pkg/EarthSciAST.jl/src/units.jl:97-101,160-161,191-192,206-207,333-337` ·
`src/validate.jl:206`

Two defects stack:

1. `validate()` calls only `validate_schema` + `validate_structural`. It **never calls the
   units engine.** `validate_model_dimensions` / `validate_file_dimensions` are exported but
   have no caller outside `test/units_test.jl`.
2. Even the direct entry point cannot fail. Every rule reports a *provable* inconsistency by
   `@warn`-ing and returning `nothing` — **the same value it returns for "unknown dimension"**
   — and `validate_equation_dimensions` (`units.jl:329-331`) treats `nothing` as consistent
   (`return true`). A violation is structurally incapable of becoming an error.

Verified: `validate_model_dimensions` returns `true` for `units_inconsistent_addition.esm`,
`units_mixed_dimensional_operations.esm`, `units_invalid_logarithm.esm`, and
`units_invalid_exponent.esm` — all fixtures the shared corpus declares **invalid**.

### C3 — TypeScript: `^` ignores its exponent; `sqrt` doesn't halve
`pkg/earthsci-ast-ts/src/units.ts:217-231, :314-319`

The `^` arm returns `finish(get(0))` — the base dimension, unchanged. The code comment is
candid about it:

> *"Preserve the base unit unchanged. Applying the exponent would require extracting the
> constant value from the second argument, which the original implementation did not attempt
> and current tests do not exercise."*

With `x → m`: `checkDimensions({op:'^', args:['x', 2]})` → `{m:1}` (want `{m:2}`).
`sqrt(A)` with `A → m^2` → `{m:2}` (want `{m:1}`). Go does `base.Power(int(exp))` correctly.

### C4 — TypeScript: the unit parser has no parentheses
`pkg/earthsci-ast-ts/src/unit-conversion.ts:149-159`

`parseUnit` splits the string on `/`, then each part on `*`, and regex-matches tokens. There is
no grouping. `parseUnit('J/(mol*K)')` returns **`{dims:{}, scale:1}` — silently dimensionless,
not an error.** Same for `cm^3/(molec*s)`. These are ordinary earth-science unit strings.

Relatedly, everything after the first `/` is treated as denominator, so `kg/m*s` parses as
`kg·m⁻¹·s⁻¹` in TS but `kg·s/m` in Go. Go's recursive-descent `ParseUnit` handles both
correctly.

**T3** (`units.ts:393-395`) compounds it: unknown/structural ops (`index`, `fn`, `aggregate`,
`const`, `makearray`, `table_lookup`, …) fall through to **dimensionless** rather than
"unknown", manufacturing *false* mismatches. Go returns `nil, nil` and skips the check.

### R6 — Rust: an integer exponent on a dimensional base is rejected
`pkg/earthsci-ast-rs/src/units.rs:304`

```rust
if let Expr::Number(n) = &op.args[1] {
```

`Expr` has a **distinct `Integer(i64)` variant** (`types.rs:249`), and JSON `2` — the natural
spelling — deserializes to `Expr::Integer(2)`, never `Expr::Number`. So the integer case falls
through to the `Err(DimensionMismatch("Cannot apply non-literal exponent to a dimensional
quantity"))` at `units.rs:313`.

Confirmed: `{"op":"^","args":["L",2]}` with `L: m` → **Err**; `{"op":"^","args":["L",2.0]}` →
Ok. The only test (`units.rs:1236`) uses `Expr::Number(3.0)` and never hits it. Consequence at
`structural.rs:113-117`: a spurious warning on every `area = L^2`, **and the equation's whole
dimension check is abandoned**, so real mismatches in that equation go unreported.

## 2. The bindings disagree about what "valid" means

No test in any binding asserts that `tests/valid/**` validates. When you actually run it:

| Binding | `tests/valid/` fixtures reported **invalid** |
|---|---|
| TypeScript | **22 of 82** |
| Go | **41 of 82** |
| Julia | accepts **15** of the `tests/invalid/` fixtures |

### T4 — TS promotes unit findings to hard errors; nobody else does
`pkg/earthsci-ast-ts/src/validate/orchestrator.ts:57-69, :270`

TS promotes unit findings to hard `unit_error` structural errors. Go, Python, Rust, and Julia
keep them as warnings. Combined with C3/C4/T3, this makes 22 of 82 valid fixtures report
`is_valid: false` **with unit codes as the only errors** (e.g. `enums_categorical_lookup.esm`,
`events_all_types.esm`). Go says `Valid=true` for those same files.

### T5 — TS rejects fractional stoichiometry; the schema and a *valid* fixture allow it
`pkg/earthsci-ast-ts/src/validate/reaction-checks.ts:64-75`

TS requires `Number.isInteger(stoichiometry)` and emits *"Stoichiometry must be a positive
integer, got 0.87"*. The schema is `{"type":"number","exclusiveMinimum":0}`, and the fixture is
literally **`tests/valid/fractional_stoichiometry.esm`**. TS: `is_valid:false`. Go:
`Valid=true`. (Python's `validation.py:333-364` implements the correct positive-finite
contract.)

### J3 — Julia's `validate()` accepts 15 invalid fixtures
`pkg/EarthSciAST.jl/src/validate.jl:244` · masked by `test/runtests.jl:182` (`@test_broken rejected`)

Beyond the 8 `units_*` fixtures (J1), Julia returns `is_valid = true` for:
`circular_coupling.esm`, `equation_count_mismatch_too_many_equations.esm`,
`undefined_parameter.esm`, `invalid_discrete_param_not_parameter.esm`,
`invalid_date_format.esm`, `invalid_url_format.esm`, `malformed_doi.esm`.

Root cause for the equation-count case: `validate_model_balance` checks only **one** direction
— a state variable with no equation. Its docstring promises "equation-unknown balance"; it
never checks for *excess* equations, nor for an equation whose LHS is not an unknown. **No
coupling-cycle check exists at all.**

### G7 — Go rejects loader-scoped references
`pkg/earthsci-ast-go/pkg/esm/validate.go:848-855` + `scoped_reference.go:50-58`

Data loaders are not added to the system namespace, so `tests/valid/data_loaders_comprehensive.esm`
fails with `[unresolved_scoped_ref] 'GEOSFP_MeteoData.u'` and `[undefined_system]
'GEOSFP_MeteoData'`. TS (`coupling-checks.ts:76-81`) includes `data_loaders`.

### Schema-level note
`esm-schema.json` declares `format: "date-time"` (×5) and `format: "uri"` (×1), but `format`
is an *annotation*, not an assertion, in JSON Schema — validators ignore it unless explicitly
configured with a format checker and the optional `rfc3339-validator` / `rfc3987` packages.
`tests/invalid/invalid_date_format.esm`, `invalid_url_format.esm`, and `malformed_doi.esm`
therefore pass schema validation everywhere. Python already concedes this with two skips
(`test_json_schema_validation.py:406,437`). These need explicit `pattern` constraints or
binding-level checks.

## 3. Hand-rolled walkers that bypass the keystone

Both Go and TS have a correct, field-preserving child-walker — and then bypass it. Every
finding in this section is the same bug wearing different clothes: *the walker only looks at
`args`.* The AST's expression-bearing sidecar fields (`expr`, `axes`, `lower`, `upper`,
`filter`, `key`, `values`, `bindings`, `regions`, `join`) are invisible.

### G2 — HIGH: Go's DAE contract silently corrupts the model
`pkg/earthsci-ast-go/pkg/esm/dae.go:160,179,183` (consequence of **G1**, `expression.go:35-59`)

`collectVariables` — backing the exported `FreeVariables` and `Contains` — hand-rolls a walk
over `Args`/`Wrt`/`Dim` only. So `Contains(rhs, "y")` is a **false negative** whenever `y`
appears inside a sidecar field.

Given `y ~ a*2.0` plus `D(z) ~ aggregate(expr: index(w,i)*y)`, `ApplyDAEContract` concludes
nothing references `y`, **deletes `y`'s defining equation**, and leaves `y` referenced inside
`expr`. The result is a surviving equation citing an undefined variable, returned with
`err = nil`. The `_ =` on both `Substitute` calls also discards a real error path.

### G4 — Go's flatten drops fields on rebuild
`pkg/earthsci-ast-go/pkg/esm/flatten.go:596`

`namespaceExpressionTree` rebuilds nodes as `ExprNode{Op, Args, Wrt, Dim}`, destroying every
other field. An event affect `{"op":"fn","name":"datetime.year","args":["t_utc"]}` becomes
`{"op":"fn","args":["M.t_utc"]}` after `Flatten` — **`name` is gone**, and the event now calls
an unnamed closed function.

### G5 — Go's `operator_compose` flatten is non-deterministic
`pkg/earthsci-ast-go/pkg/esm/flatten.go:643-655`

The `translate` map is applied by **iterating a Go map** (randomized order) with sequential
in-place text replacement, so renames chain. Flattening the *same file* 20× produced two
different systems: `"B.r + B.r"` ×18 and `"A.q + B.r"` ×2.

### G3 / G11 / T7 / T8 / T9 / T10 — the rest of the cluster
- **G3** (`substitute.go:80-92`, `decode.go:48`) — `UnmarshalExpression` only normalizes `args`;
  every other expression field stays a raw `map[string]any`, and `substituteRec`'s `default:`
  returns raw maps unchanged. `Substitute(aggregate{expr:{*:[index(w,i),"a"]}}, {"a":99})`
  leaves `a` **unsubstituted** — while the identical node inside `args` substitutes fine.
- **G11** (`graph.go:487-513`) — a *third* hand-rolled walker, `args`-only, with no `*ExprNode`
  case at all → dependency edges through `expr`/`axes` vanish silently.
- **T7** (`expression.ts:35-52,60-72,80-91`) — `freeVariables`/`contains`/`freeParameters` walk
  `args` only, though the module's own header says to use `mapChildren`.
  `freeVariables(table_lookup{axes:{temp:'T_air'}})` → `[]`. `graph.ts:574+` builds the
  dependency DAG from this, so **edges silently vanish** there too.
- **T8** (`flatten.ts:430-474`) — `expressionNodeToString`'s generic fallback is
  `` `${op}(${args})` ``, so sidecar-bearing nodes lose their entire body:
  `{op:'table_lookup', table:'kT', args:[], axes:{temp:'T_air'}}` stringifies to
  **`"table_lookup()"`**.
- **T9** (`coupling-imports.ts:101-115, :204-217`) — `rewriteExprInPlace` rebinds refs only in
  `args` and `apply_expression_template.bindings`; a library-role ref inside
  `expr`/`axes`/`lower`/`filter` survives `bind`, producing a **dangling role reference**. The
  read-only twin `forEachExprRef` has the same gap, so validation doesn't flag it either.
- **T10** (`substitute.ts:180-261`) — `substituteInModel` omits `initialization_equations` and
  `guesses`, both Expression sites → dangling refs after a rename.

**Recommended fix:** delete the hand-rolled walkers and route every one of these through
`mapExprChildren` / `mapChildren`. Julia already did exactly this (`_referenced_var_names` now
routes through `child_exprs`), and its walker layer came out of this audit clean.

## 4. Rust — no operator-arity validation anywhere

The schema sets `args.minItems: 0` and **nothing in the crate validates arity** (no op
registry; neither `validate.rs` nor `structural.rs` checks it). Four findings share this root
cause. All were reproduced empirically.

- **R1 · HIGH** (`simulate_array/eval.rs:115`, `:571-575`) — `atan2` dispatches to `eval_binary`,
  which indexes `args[0]`/`args[1]` with **no arity check**, unlike the comparison arm directly
  above it. `{"op":"atan2","args":[1.0]}` → **panic** (index out of bounds). Same for `args: []`.
- **R2 · HIGH** (`simulate_array/vectorized.rs:552`, `:559-562`) — the vectorized
  `eval_vec_op` indexes `node.args[0]` with no emptiness guard, while the oracle guards both.
  `{"op":"neg","args":[]}` → **panic** on the vectorized path, `NaN`/`-0.0` on the oracle path.
- **R3 · HIGH — the two evaluators compute different values.** `fold_scalar`
  (`eval.rs:307-351`) returns `NaN` for `-`/`/`/`^` unless `len == 2` and for `min`/`max` unless
  `len >= 2`; the vectorized path (`vectorized.rs:545-558`) **left-folds any arity**:

  | node | oracle | vectorized |
  |---|---|---|
  | `-(3,1,1)` | `NaN` | `1.0` |
  | `/(8,2,2)` | `NaN` | `2.0` |
  | `min(5)` | `NaN` | `5.0` |
  | `and(5)` | `1.0` | `5.0` (raw, not a boolean) |

  Worse, **the oracle is inconsistent with itself**: `eval_arith`'s all-scalar guard
  (`eval.rs:284`) routes to `fold_scalar`, but any *array* operand routes to the left-folding
  `combine`. So `{"op":"-","args":["u",1,1]}` yields `u−2` when `u` is an array and `NaN` when
  it is a scalar — three semantics for one AST node, depending on operand type *and* on whether
  the enclosing body happens to vectorize.
- **R5 · MEDIUM-HIGH** (`eval.rs:1174-1191`) — `makearray` takes `ndim` from `regions[0].len()`
  then indexes `lo[d]`/`hi[d]` on every region **without checking its own length** (the
  vectorized twin *does* check). `regions: [[[1,3]], [[1,3],[1,2]]]` → **panic**. A reversed
  region `[[[5,2]]]` gives extent `-2`, and `as usize` → `ArrayD::zeros` capacity-overflow
  **panic**.

Two more Rust findings, not arity-related:

- **R4 · HIGH** — an array-valued `filter` is a genuine per-cell mask on the vectorized path
  (`vectorized.rs:216-242`) but **"exclude everything"** on the oracle (`eval.rs:938-943`, where
  a non-scalar filter coerces via `as_scalar().unwrap_or(0.0)` → `0.0` → excluded). With
  `mask = [1,0,1]` and body ≡ `10`: vectorizable body → `[10,0,10]`; the same filter with a
  `reshape` in the body (which bails to the oracle) → `[0,0,0]`. **Which answer you get depends
  solely on incidental vectorizability.** The doc-comments on the vectorized path advertise
  exactly this use ("a regrid `overlap > 0` filter", "a per-cell fuel gate").
- **R7 · MEDIUM** (`eval.rs:198`) — `eval_op`'s `_ => Value::Scalar(f64::NAN)` catch-all
  silently NaNs every closed-core op with no evaluator arm: `argmin`, `argmax`, `skolem`,
  `rank`, `true`, `integral`, un-lowered `table_lookup`/`enum`/`apply_expression_template` —
  **and any typo'd op name** (`"expp"` → NaN, not an error).

**One fix closes four of these:** an op-arity table checked at parse or at `from_model`.

## 5. Simulation / evaluation semantics

- **P2 · HIGH — Python `simulate()` silently ignores `flat.discrete_events`**
  (`simulation_scalar.py:156-171`). `flatten()` populates them (`flatten.py:1324`), but the
  live simulate paths read only `flat.continuous_events`; `simulate_with_discrete_events`
  (`simulation_legacy.py:459`) exists and nothing routes to it. On
  `tests/simulation/periodic_dosing.esm` the dose never applies: `drug_amount` is `0.0` at
  `t=8.1` (expected `99.005`) and at `t=16.1` (expected `143.491`), with
  `result.success == True`. **Silent wrong answer, not an error.**

- **P3 · HIGH — a terminal continuous event makes Python extrapolate past the integration end**
  (`simulation_array.py:88-97`). `_create_event_functions` sets `terminal = True` and attaches
  an `.affects` list, but `_simulate_scalar` calls `solve_ivp` **once** and never applies the
  affects or re-integrates — so continuous-event state resets are dead. `_densify_solution` then
  clips its output grid to the **requested** `tspan` rather than the **achieved** span
  `[sol.t[0], sol.t[-1]]`, and SciPy's `OdeSolution` happily extrapolates the last interpolant
  polynomial past the terminal stop. On `bouncing_ball.esm` the solver terminates at `t=1.4278`
  but the result reports `t` out to 10 with `height = -480.5` — the ball 480 m below the floor,
  `success = True`. **86% of the returned trajectory is polynomial garbage.** This bites *any*
  model with a terminal event.

- **T6 · HIGH — TS codegen evaluates `ifelse`/`and`/`or` eagerly** (`codegen.ts:202-204`:
  `node.args.map(evalExprNode)` runs *before* `info.evaluate(args)`). The untaken branch runs and
  its domain error propagates, breaking the canonical guard idiom:
  `ifelse(x>0, log(x), 0)` at `x=-1` **throws**; `ifelse(x!=0, 1/x, 0)` at `x=0` **throws**;
  `or(1, unbound)` **throws**. (Julia's `_eval_node_op` and `_oop_eval_op` are both correctly
  lazy — this is exactly the bug class Julia's CSE finding #1 was about.)

- **G6 · HIGH — Go's evaluator implements none of the closed comparison/boolean tier**
  (`expression.go:293-380`, `:494-497`). `<`, `>`, `>=`, `<=`, `==`, `!=`, `and`, `or`, `not`,
  `ifelse`, `true`, `Pre` all return `unknown operation`. A discrete-event trigger condition is
  unevaluable in Go. TS implements all of them. Open-tier ops also get a plain `fmt.Errorf`
  rather than the spec-pinned `unlowered_operator` code.

- **J2 · HIGH — Julia's MTK continuous events are broken; every model with one fails to build**
  (`ext/mtk_ext/variables.jl:288-289`). The 2-arg
  `SymbolicContinuousCallback(conds, affects)` constructor no longer exists in current MTK.
  Reproduced: `ModelingToolkit.System(bouncing_ball_model)` →
  `MethodError: no method matching SymbolicContinuousCallback(::Num, ::Vector{Symbolics.Equation})`.
  The `continuous_events` feature of the MTK export is **entirely non-functional** — and
  invisible, because `test/tests_blocks_execution_test.jl:163` replaces the fixture with
  `@test_skip false`.

- **P8 · MEDIUM — Python's planar polygon clip silently under-counts a non-convex clip operand**
  (`geometry.py:148-190`). `_planar_clip` is Sutherland–Hodgman, valid only for a **convex** clip
  ring. The docstring admits a non-convex operand "would silently give the convex-edge
  intersection and is out of contract" — but **there is no guard**, and `intersect_polygon` is a
  public evaluable-core op the schema places no convexity restriction on. Subject = 3×3 square,
  clip = L-shape → reported area **1.0**, true overlap **3.0**. For a *conservative* regridding
  kernel whose entire contract is mass conservation, silently under-counting overlap area is a
  correctness hazard, and the realistic trigger is the ordinary earth-science case: regridding
  onto a non-convex basin, coastline, or administrative polygon.

## 6. Julia codegen — the recent fix pass holds, with one new bug

I attacked all seven findings from `julia_iip_cse_cadence_audit_2026-07-14.md`. **All seven
fixes hold.** Details worth recording:

- **#1** — `_cse_arg_conditional` (`compile.jl:653`) enumerates *exactly* the lazy arms of
  `_eval_node_op`; the `total>=2 && uncond>=1` rule is correct. Bonus: `_oop_eval_op` was also
  made lazy (`oop.jl:360-385`), closing a scalar/oop divergence the audit didn't ask about.
- **#4b/c/d** (`invariant_share.jl`) — `_node_vn` is total over every `_Node` field and fails
  closed; the topological-order argument for new slots is sound.
- **#4a** (`vec_share.jl`) — the pass-through set really is exactly `{1-ary +, 1-ary *, Pre}`,
  and `_vk_hoistable` refuses exactly those. In-degree is counted per *edge* over the DAG, which
  makes `x*x` correct.
- **#5** (const tier) — verified empirically: `n_cse_slots=5 / n_const=5 / n_dyn=0` on the
  Arrhenius model, correct values across alternating `p1 → p2 → p1 → p2 → p2` sweeps.
- **#7** — content-keyed hash plus a `_check_fn_group_specs` re-check.

A sweep for "the next #7" (an incomplete cache key or structural signature) turned up four
near-misses, **all of which are actually safe** — but three rest on undocumented invariants
worth pinning with a comment or an assertion: `_struct_sig!`'s `_NK_CONTRACTION` arm omits
`n.literal` (safe only because groups never span semirings); `_cse_rebuild` (`compile.jl:836`)
drops every non-`args` field (safe only because the 16 ops `_compile_op` special-cases are
*exactly* `_CSE_OPAQUE_OPS` — I diffed the lists); `_branch_key!` (`stencil.jl:294`) doesn't
walk `filter`/`key`/`ranges` (safe only because `_stencilize` throws `_StencilFallback` on
every construct that can carry them).

### J4 · MEDIUM (NEW) — the CSE key collapses a non-value-preserving fold
`src/tree_walk/compile.jl:635` (`_cse_key`) · `src/canonicalize.jl:106-116`, `:162-164`

`canonicalize` folds `x*0 → 0` and `0/x → 0` **regardless of `x`**, so both emit the canonical
key `"0.0"`. `_cse_count!` sees two occurrences of one key, hoists a single slot, and compiles
its definition from the **first-seen** occurrence — so every other `"0.0"`-keyed expression
reads a value computed from a *different* expression.

Reproduced end-to-end:

```
D(x) = (1/z)*0                  alone  ->  n_cse_slots=0,  du[x] = NaN   (correct: Inf*0)
D(y) = w*0 ; D(x) = (1/z)*0            ->  n_cse_slots=1,  du[x] = 0.0   (WRONG)
```

`D(x)` is byte-identical between the two models. **Adding an unrelated equation turned its
`NaN` into `0.0`**; reversing the equation order makes the healthy `D(y) = w*0` produce `NaN`
instead.

This shows finding **#2's "documented + pinned" status is incomplete**. The pin
(`compile.jl:421-445`) characterizes the failure mode as float *reassociation* — "normally
sub-ulp; catastrophic cancellation can amplify it". That does not cover this: the root cause is
a different canonicalize rule, and the error is **unbounded** (`NaN ↔ 0.0`), not sub-ulp.
Fail-closing `_cse_key` on any expression whose canonical form is a bare literal closes it.

### J5 · MEDIUM-HIGH (re-confirms the existing `@test_broken`) — Reactant/XLA + `param_arrays`
`test/reactant_oop_test.jl:264`

A `_VK_PGATHER`'s payload is an aliased host `Vector{Float64}`; to the XLA tracer it is a
*constant*, so the compiled program bakes in the buffer's compile-time contents and never sees
the in-place refresh a data-refresh callback performs. Not a crash, not a NaN — **the same
numbers forever**, off by the full magnitude of the forcing update. Nothing prevents a user from
`@compile`-ing a model built with `param_arrays`; a build-time refusal would be cheap.

## 7. Round-trip and serialization fidelity

- **G8 · HIGH** (`types.go:15-104`) — Go's `ExprNode` has **no `id` and no `expect_cadence`
  field**, both schema-declared on `ExpressionNode`. `Load` → `Serialize` of
  `{"op":"*","args":["a",2.0],"id":"my_node_id","expect_cadence":"const"}` emits
  `{"args":["a",2.0],"op":"*"}` — **silently dropped**. `id` is the referent of a derived index
  set (`index_sets.<x>.from_faq`), so a Go round-trip **breaks that linkage**. TS preserves both.
- **G9 · MEDIUM** (`canonicalize.go:472-495`) — `exprNodeHasNonEmissibleField` is a **deny-list**
  that omits `Label`, violating its own stated "every OTHER field is listed here" invariant.
  `CanonicalJSON({"op":"skolem","args":["a","b"],"label":"edge"})` drops the label; TS's
  allow-list throws `E_CANONICAL_UNSUPPORTED_FIELD`. This is the argument for allow-lists over
  deny-lists, and Julia's `_NON_EMISSIBLE_FIELDS` (derived fail-closed from
  `fieldnames(OpExpr)`) is the model to copy.
- **T11 · MEDIUM** (`numeric-literal.ts:317-321` vs `:433-436`) — the parser tags the integer
  token `-0` as `{kind:'int'}`, but `formatIntToken` throws
  `TypeError: int NumericLiteral cannot hold -0`. `losslessJsonParse('{"v": -0}')` succeeds and
  `losslessJsonStringify` on the result **throws** — the parser manufactures a value its own
  serializer rejects.
- **G12 · MEDIUM** (`metaparams.go:86-93`) — `metaSubstSkipKeys` omits `"op"`, so a
  metaparameter named `max` rewrites `{"op":"max"}` to `{"op":3}` and `LoadString` dies with a
  raw unmarshal error instead of a diagnostic.
- **G13 · MEDIUM** (`flatten.go:719-728`, `:774-776`) — `factor` is spliced as **unparenthesized
  text**, so `2*B.y^2` reparses as `2*(B.y²)` when the correct reading is `(2*B.y)^2`; and
  connector target matching uses `strings.Contains`, so `to: "A.v"` also matches `D(A.v2, t)`.
- **G15 · MEDIUM** (`lower_enums.go:40-69,163-165`) — the enum-lowering walk skips
  `discrete_events`/`continuous_events` entirely and its `default:` returns raw maps, so an
  `enum` in a sidecar **survives `LowerEnums`** (contract violated) and `unknown_enum` is dead
  in those positions.
- **G14 · MEDIUM** (`validate.go:138`) — `ValidateFile.IsValid` ignores `Level`, so a
  warning-only finding makes `IsValid=false` while `ValidateStructural` says valid.

## 8. The conformance harness cannot catch any of the above

This is the finding that makes all the others *durable*.

### C1 · CRITICAL — the harness exits 0 with languages failing
`scripts/test-conformance.sh:689-848`

`failed_languages` is printed via `error()` and then **never fails the script**. `main`
proceeds whenever ≥2 languages succeeded and ends on a `success` echo. So
`./scripts/test-conformance.sh` — the command `CLAUDE.md` instructs developers to run — prints
**"Cross-language conformance testing completed successfully!"** and exits 0 with Julia, Rust,
and Go all failing.

### C2 · CRITICAL — the comparator passes at 70% divergence, on two vacuous categories
`scripts/compare-conformance-outputs.py:437-442, :462`

`return 0 if overall_status in ["PASS","WARN"]`, where `PASS ≥ 0.90` and `WARN ≥ 0.70`. **Up to
30% cross-language divergence exits 0.** Demonstrated: 2 of 10 valid fixtures where Julia says
valid and Python says invalid → `Overall status: PASS, 93.33%`, exit 0.

It is worse than the threshold suggests, because **two of the three compared categories are
structurally vacuous** and *dilute the one real category*:

- Rust emits literal `"display_results": {}, "substitution_results": {}` (`bin/esm.rs:3189-3190`).
- Julia/TS only populate those when a fixture has top-level `chemical_formulas` / `expressions`
  / `tests` keys — and **no fixture in `tests/display/` has those keys** (they are top-level
  *lists*, or use `test_cases`).
- The comparator takes `reference_lang = list(file_results)[0]` = julia (empty) → zero
  divergences → every file counted "consistent".

So display and substitution are pinned at 1.00 forever. That is how the 20% validation
divergence above still averaged to a PASS.

### F4 · HIGH — the comparator never checks the *expected* outcome
`compare-conformance-outputs.py:107-137`

It compares languages **to each other**, never to the fixture's declared outcome. The
`is_expected_error` field every producer emits is **never read**. A `tests/valid/*.esm`
rejected by *all* bindings, or a `tests/invalid/*.esm` accepted by *all* bindings, is
"consistent" → PASS. A shared regression ships green.

The same gap exists inside the bindings' own suites: `conformance.test.ts:224-256` loads
`tests/invalid/expected_errors.json` but **only reads its `resolver_only` flag** — it never
compares actual codes/paths against expected. TS emits `unit_error @ /models/BadUnitsModel`
where the fixture pins `unit_inconsistency @ /models/BadUnitsModel/equations/0`, and CI is
green.

### F5 · HIGH — 69 fixtures are never swept
All four producers (`run-julia-conformance.jl:56`, `run-python-conformance.py:87,112`,
`run-typescript-conformance.js:65,96`, `bin/esm.rs:3053,3085`) sweep `tests/valid` and
`tests/invalid` **non-recursively**. That misses 34 fixtures in
`valid/{aggregate,cadence,geometry,initial_conditions,scalar_leaves}` and 35 in
`invalid/{aggregate,template_imports,coupling,geometry}` — including the *entire* aggregate and
template-import corpora.

## Suggested order of work

The ordering is chosen so that each step makes the *next* step's bugs visible rather than
letting them hide again.

1. **Make the harness able to fail** (C1, C2, F4). `exit 1` on any failed language; drop the
   0.9/0.7 thresholds to "any divergence fails"; assert each fixture against its **expected**
   outcome (`is_expected_error`, `expected_errors.json` codes/paths) rather than mere
   inter-language agreement. Until this lands, every fix below can silently regress.
2. **Add the assertion nobody wrote**: `tests/valid/**` must validate in all five bindings, and
   `tests/invalid/**` must be rejected. That single test turns the 22/82, 41/82, and 15-accepted
   numbers into a concrete, shrinking work-list.
3. **Fix or delete the vacuous comparison categories** (C2) — right now display and substitution
   manufacture a false 100% that masks real validation divergence.
4. **Units**, in dependency order: Python C5 first (it makes P6/P7 visible), then TS C3/C4/T3,
   then Rust R6, then Julia J1. Settle the warning-vs-error question (T4) **once, in the spec**,
   before fixing the bindings — otherwise they will re-diverge.
5. **Delete the hand-rolled walkers** (G1–G4, G11, T7–T10) and route everything through
   `mapExprChildren` / `mapChildren`. G2 corrupts models and should jump the queue.
6. **Add an op-arity table to Rust** (closes R1, R2, R3, R5 at once) and reconcile
   `fold_scalar` with `apply_binary` so the two evaluators agree.
7. **Python simulation**: discrete events (P2) and the terminal-event extrapolation (P3).
8. **Julia**: fail-close `_cse_key` on bare-literal canonical forms (J4); fix or fail loudly on
   `SymbolicContinuousCallback` (J2).
9. **Repair the skip mechanism**: convert Python's imperative `xfail` to
   `@pytest.mark.xfail(strict=True)`, delete the 13 stale entries, replace Julia's
   `@test_skip false` markers, and either re-file the ~20 dangling bead IDs or stop citing bead
   IDs in skips that no tracker backs.
